task1

针对前期作业中选定的业务背景，请完成如下数据库应用系统规划训练

  1. 给你所规划的数据库应用系统起一个系统名称
  2. 尝试调研分析或自主规划设计该系统的业主企业或组织机构的组织架构图，并说明该系统涉及企业或组织机构中哪些相关业务部门。
  3. 调研分析企业相关部门中的跟该系统有关的各种用户或企业外的用户，用文字描述用户使用系统开展业务的场景（例如，租车客户将在特定手机上打开APP，用户在界面上点击….）
  4. 规划系统的性能指标，如并发用户数、用户数、核心业务响应时间等。
  5. 说明你所规划的系统的战略地位，例如系统服务于公司以什么方式赢得客户、获得直接或间接收益，获得市场地位。
  6. 尝试说明建设该系统可能涉及的投资和运营成本，分析可能获得的收益。
  7. 结合系统业务功能与性能规划，确定初步的技术选型规划，大致分析该系统在技术上的可行性。
    将以上内容形成系统规划与可行性分析报告，找两位同学当业务专家帮你论证方案的可行性，并在报告上列出专家姓名、专家论证建议和结论。

感谢提供SQL脚本，以下是对你的数据库系统（基于朋友圈/社交动态平台）的初步分析和规划。我们将基于你建库脚本中定义的几张表（用户 user、朋友圈动态 moment、多媒体内容 media 等）来构建完整的系统规划、需求分析与设计文档。

1 系统名称

Wavelet Matrix

2 组织机构与业务部门

组织架构示意：

涉及部门

  • 技术部：实现系统后端逻辑与数据库管理；
  • 审核部：处理动态中的违规内容；
  • 客服：处理用户反馈、封号等问题；
  • 产品部：定义系统业务逻辑与用户体验流程。

3 用户与业务场景

  • 普通用户：打开移动APP或Web平台 → 查看好友动态 → 发布图文/视频 → 点赞评论。
  • 审核：登录管理后台 → 审查被举报内容 → 删除/警告处理。
  • 管理：管理用户信息，处理系统配置。

4 系统性能指标

  • 并发用户数：支持 1 万用户同时在线
  • 总用户容量：100 万用户
  • 核心业务响应时间：
    • 发布动态 < 1 秒
    • 点赞/评论 < 300 毫秒

5 系统战略地位

该系统为公司主要流量入口与用户黏性载体：

  • 提高用户日活跃度（DAU）
  • 促进广告投放与品牌合作
  • 通过付费功能（置顶动态、表情包等）实现营收

6 投资与收益

  • 投资成本：

    • 初期开发人力约 80 万元
    • 云服务与数据库托管约 5 万元/年

  • 运营成本：

    • 内容审核与客服人力

  • 收益来源：

    • 广告展示
    • 用户增值服务
    • 数据分析服务

7 技术选型与可行性

技术栈规划：

  • 后端：Node.js / Python Flask + MySQL
  • 前端：React + 微信小程序
  • 数据库：MySQL，支持全文索引和事务
  • 存储：对象存储服务

可行性分析：

  • 数据模型简单、关系清晰，适合用关系型数据库；
  • 高并发读写可通过缓存层（如Redis）优化；
  • 多媒体内容独立存储，利于扩展与CDN分发。

好的，以下是第二部分：数据库应用系统需求规格说明书，将涵盖UML相关概念与图示，以及对你的“社交动态平台系统”做详细需求分析和建模。

1 UML基础与用例分析

UML 简介

UML（统一建模语言）是一种用于系统分析与设计的图形化建模语言，常用于表示系统结构、功能与行为。

用例（Use Case）

表示系统能完成的某个功能（比如“发布动态”）。

用况场景（Use Case Scenario）

描述用例实际执行过程中的具体流程。

用例图（Use Case Diagram）

展示用户与系统交互的功能结构。

泳道图（Swimlane Diagram）

用来清晰展示多个用户（或角色）之间业务流程的分工与协作。

数据流图（DFD）

用于表示信息在系统中如何流动与处理。

extask1

我们来逐步完成题目中的两个任务：

第7题：学院的E-R图设计

实体：

  1. 系：系编号、系名
  2. 班级：班级编号、班级名
  3. 教研室：教研室编号、教研室名
  4. 教员：职工号、姓名、职称
  5. 学生：学号、姓名、学历
  6. 课程：课程编号、课程名
  7. 成绩：成绩（联系的属性）

联系：

  • 系与班级：一个系有多个班级（1:N）
  • 系与教研室：一个系有多个教研室（1:N）
  • 教研室与教员：一个教研室有多个教员（1:N）
  • 教员与研究生（学生）：教授和副教授指导多个研究生（1:N，可设为“指导”）
  • 班级与学生：一个班级有多个学生（1:N）
  • 学生与课程：多对多，学生选修课程，产生一个成绩（M:N，有联系“选修”）

第8题：工厂的E-R图设计

实体：

  1. 产品：产品号，产品名
  2. 零件：零件号，零件名
  3. 原材料：原材料号，原材料名，类别
  4. 仓库：仓库号，仓库名

联系：

  • 产品由多个零件组成（M:N）
  • 零件由不同原材料组成（M:N）
  • 零件归属于某个产品，存储在某个仓库（多对一）
  • 原材料按类别放入不同仓库（多对一）

第7题关系模型：

  1. 系(系编号 PK, 系名)
  2. 教研室(教研室编号 PK, 教研室名, 系编号 FK)
  3. 老师(职工号 PK, 姓名, 职称, 教研室编号 FK)
  4. 班级(班级编号 PK, 班级名, 系编号 FK)
  5. 学生(学号 PK, 姓名, 学历, 班级编号 FK)
  6. 职工(职工号 FK, 学号 FK) — 教员指导学生（职工号限制为教授/副教授）
  7. 课程(课程编号 PK, 课程名)
  8. 选课(学号 FK, 课程编号 FK, 成绩) — 学生选课成绩，多对多关系
  • 所有基本关系均为3NF，若每个非主属性只依赖于主键。
  • 选课关系不满足BCNF（因为学号, 课程编号 → 成绩，但课程编号 → 成绩 不成立）=> 满足BCNF。

第8题关系模型：

  1. 产品(产品号 PK, 产品名)
  2. 零件(零件号 PK, 零件名)
  3. 原材料(原材料号 PK, 原材料名, 类别)
  4. 仓库(仓库号 PK, 仓库名)
  5. 产品零件(产品号 FK, 零件号 FK) — M:N 关系
  6. 零件原材料(零件号 FK, 原材料号 FK) — M:N 关系
  7. 零件仓库(零件号 PK, 仓库号 FK) — 零件存在哪个仓库（一对多）
  8. 原料仓库(原材料号 PK, 仓库号 FK) — 原材料存在哪个仓库（一对多）
  • 所有关系为3NF
  • M:N 关系转化后，每个表的主键是复合主键，满足BCNF
